# MySQL 索引

## 索引概述

索引是一种特殊的数据结构，用于快速查找表中的数据。类似于书的目录，索引可以大幅提升查询速度，但会增加存储空间和写操作的开销。

### 索引的优缺点

| 优点 | 缺点 |
|------|------|
| 加速查询 | 占用磁盘空间 |
| 减少 I/O 操作 | 增删改时需要维护索引，增加开销 |
| 帮助服务器避免排序和临时表 | 过多索引会增加优化器选择难度 |

## MySQL 索引类型

### 1. B+Tree 索引（默认）

InnoDB 存储引擎使用 B+Tree 作为索引结构，所有数据都存储在叶子节点。

```
        [50]
       /    \
   [20,30]  [70,90]
   / | \    / | \
 ...       ...
 
 B+Tree 特点：
 - 所有叶子节点在同一层（平衡）
 - 叶子节点有序链表连接（范围查询快）
 - 非叶子节点只存储索引，不存储数据
```

### 2. 哈希索引

适用于精确匹配查询，Memory 存储引擎支持。

```sql
-- 创建哈希索引（Memory 存储引擎）
CREATE TABLE t1 (
    id INT,
    name VARCHAR(50),
    INDEX idx_name USING HASH (name)
) ENGINE=MEMORY;
```

### 3. 主键索引 vs 普通索引

```sql
-- 主键索引：PRIMARY KEY，自动唯一非空
CREATE TABLE users (
    id INT PRIMARY KEY,  -- 主键索引
    ...
);

-- 普通索引：INDEX 或 KEY
CREATE TABLE orders (
    id INT PRIMARY KEY,
    user_id INT,
    INDEX idx_user_id (user_id)  -- 普通索引
);
```

### 4. 唯一索引

```sql
-- 唯一索引：值必须唯一，可以为空
CREATE TABLE users (
    email VARCHAR(100),
    UNIQUE INDEX idx_email (email)
);

-- 复合唯一索引
CREATE TABLE user_roles (
    user_id INT,
    role_id INT,
    UNIQUE INDEX idx_user_role (user_id, role_id)
);
```

### 5. 全文索引

```sql
-- 全文索引：用于文本搜索
CREATE TABLE articles (
    id INT PRIMARY KEY,
    title VARCHAR(200),
    content TEXT,
    FULLTEXT INDEX idx_content (title, content)
);

-- 使用全文索引
SELECT * FROM articles 
WHERE MATCH(title, content) AGAINST('keyword');
```

### 6. 前缀索引

```sql
-- 对字符串前N个字符建索引
ALTER TABLE users ADD INDEX idx_email (email(10));

-- 适用场景：字符串很长，但前面部分就能区分
-- 注意：前缀长度选择要合适，太短区分度低，太长失去意义
```

## 创建索引

### 建表时创建

```sql
CREATE TABLE users (
    id INT PRIMARY KEY AUTO_INCREMENT,
    username VARCHAR(50) NOT NULL,
    email VARCHAR(100) NOT NULL,
    age INT,
    created_at TIMESTAMP,
    -- 普通索引
    INDEX idx_username (username),
    -- 唯一索引
    UNIQUE INDEX idx_email (email),
    -- 复合索引
    INDEX idx_age_created (age, created_at)
) ENGINE=InnoDB;
```

### 已有表添加索引

```sql
-- 添加普通索引
ALTER TABLE users ADD INDEX idx_username (username);

-- 添加唯一索引
ALTER TABLE users ADD UNIQUE INDEX idx_email (email);

-- 添加复合索引
ALTER TABLE users ADD INDEX idx_age_created (age, created_at DESC);

-- 添加全文索引
ALTER TABLE articles ADD FULLTEXT INDEX idx_content (content);
```

### CREATE INDEX 语法

```sql
-- 基本语法
CREATE INDEX idx_name ON table_name (column);

-- 唯一索引
CREATE UNIQUE INDEX idx_email ON users (email);

-- 复合索引
CREATE INDEX idx_age_created ON users (age, created_at DESC);

-- 指定索引类型
CREATE INDEX idx_name ON users (name) USING BTREE;  -- 默认
```

## 删除索引

```sql
-- 删除索引
DROP INDEX idx_username ON users;

-- 通过 ALTER TABLE 删除
ALTER TABLE users DROP INDEX idx_email;
ALTER TABLE users DROP PRIMARY KEY;  -- 删除主键
```

## 查看索引

```sql
-- 查看表的所有索引
SHOW INDEX FROM users;

-- 查看建表语句（包含索引信息）
SHOW CREATE TABLE users\G

-- MySQL 8.0 新增：可视化查看
SHOW INDEX FROM users FROM blog_db;
```

## 复合索引（最左前缀原则）

复合索引遵循最左前缀原则：索引 (A, B, C) 可以用于查找：

```sql
-- 可以使用索引的情况
SELECT * FROM users WHERE age = 25;           -- ❌ 没用索引（没从最左列开始）
SELECT * FROM users WHERE username = 'john';   -- ✅ 使用索引
SELECT * FROM users WHERE username = 'john' AND age = 25;  -- ✅ 使用索引
SELECT * FROM users WHERE username = 'john' AND age = 25 AND created_at > '2024-01-01';  -- ✅ 使用索引

-- 范围查询会阻断后续列使用索引
SELECT * FROM users WHERE username = 'john' AND age > 25;  -- ⚠️ age 之后的列无法使用索引
```

### 索引列顺序选择

根据查询频率和选择性选择顺序：

1. **等值查询列优先**：=, IN
2. **范围查询列靠后**：<, >, BETWEEN
3. **高选择性列靠前**：区分度高的列

## 索引优化技巧

### 1. 避免全表扫描

```sql
-- ❌ 不好：email 没索引，全表扫描
SELECT * FROM users WHERE email = 'test@example.com';

-- ✅ 好：username 有索引
SELECT * FROM users WHERE username = 'john';
```

### 2. 覆盖索引

查询的所有列都在索引中，无需回表：

```sql
-- 创建覆盖索引
ALTER TABLE users ADD INDEX idx_cover (username, email, age);

--  explain 查看是否 Using index
EXPLAIN SELECT username, email, age FROM users WHERE username = 'john';
-- Extra: Using index （覆盖索引，无需回表）
```

### 3. LIKE 查询优化

```sql
-- ❌ '%xxx' 开头无法使用索引
SELECT * FROM users WHERE email LIKE '%@gmail.com';

-- ✅ 'xxx%' 开头可以使用索引
SELECT * FROM users WHERE username LIKE 'john%';
```

### 4. 避免函数和运算

```sql
-- ❌ 对索引列使用函数，无法使用索引
SELECT * FROM users WHERE YEAR(created_at) = 2024;

-- ✅ 使用范围查询
SELECT * FROM users WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01';

-- ❌ 运算导致无法使用索引
SELECT * FROM users WHERE age + 1 = 26;

-- ✅ 正确的写法
SELECT * FROM users WHERE age = 25;
```

### 5. OR 条件优化

```sql
-- ❌ OR 条件可能导致全表扫描
SELECT * FROM users WHERE username = 'john' OR email = 'john@example.com';
-- 需要两列都有索引才行

-- ✅ 使用 UNION 替代
SELECT * FROM users WHERE username = 'john'
UNION
SELECT * FROM users WHERE email = 'john@example.com';
```

## EXPLAIN 分析查询

```sql
EXPLAIN SELECT * FROM users WHERE username = 'john'\G

-- 关键列解释：
-- type: 连接类型（system > const > eq_ref > ref > range > index > ALL）
--       最好的是 const/eq_ref，最差是 ALL（全表扫描）
-- key: 实际使用的索引
-- rows: 预计扫描的行数
-- Extra: 额外信息（Using index 表示覆盖索引）
```

### 优化示例

```sql
-- 原始查询：全表扫描
SELECT * FROM users WHERE email LIKE '%@gmail.com';

-- 优化方案1：如果是前缀匹配，改用后缀
SELECT * FROM users WHERE email LIKE 'user%@gmail.com';

-- 优化方案2：添加反向索引列
ALTER TABLE users ADD COLUMN email_domain VARCHAR(50);
UPDATE users SET email_domain = SUBSTRING_INDEX(email, '@', -1);
CREATE INDEX idx_domain ON users(email_domain);
SELECT * FROM users WHERE email_domain = 'gmail.com';
```

## 索引失效场景

| 场景 | 示例 | 原因 |
|------|------|------|
| 函数运算 | `WHERE YEAR(d) = 2024` | 索引列参与计算 |
| 类型转换 | `WHERE id = '123'`（id是INT） | 类型不匹配 |
| OR 混用 | `WHERE name = 'a' OR age > 20` | 缺少索引列 |
| LIKE 前缀 | `WHERE name LIKE '%abc'` | 无法确定前缀 |
| 范围在后 | `WHERE a = 1 AND b > 2 AND c = 3` | b 是范围，阻断 c |
| IS NOT NULL | `WHERE email IS NOT NULL` | 部分存储引擎不支持 |

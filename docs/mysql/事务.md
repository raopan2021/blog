# MySQL 事务

## 什么是事务

事务是数据库操作的逻辑单元，一个事务内的所有操作要么全部成功，要么全部失败回滚。事务确保数据的完整性和一致性。

### 事务的 ACID 特性

| 特性 | 说明 |
|------|------|
| **Atomic（原子性）** | 事务是最小执行单位，不可分割。所有操作要么全部成功，要么全部失败。 |
| **Consistency（一致性）** | 事务执行前后，数据库状态保持一致。例如转账前后，总金额不变。 |
| **Isolation（隔离性）** | 并发执行的事务相互隔离，不互相干扰。 |
| **Durability（持久性）** | 事务提交后，对数据库的修改永久保存，即使系统崩溃也不会丢失。 |

### 事务的典型场景：转账

```sql
-- 转账场景：A 向 B 转账 100 元
START TRANSACTION;

UPDATE accounts SET balance = balance - 100 WHERE user_id = 1;  -- A 扣款
UPDATE accounts SET balance = balance + 100 WHERE user_id = 2;  -- B 收款

-- 检查结果，如果有问题则回滚
COMMIT;  -- 提交事务
-- ROLLBACK;  -- 如果出错，执行回滚
```

## 事务控制语句

```sql
-- 开启事务
START TRANSACTION;
-- 或
BEGIN;

-- 提交事务
COMMIT;

-- 回滚事务
ROLLBACK;

-- 设置保存点
SAVEPOINT savepoint_name;

-- 回滚到保存点
ROLLBACK TO SAVEPOINT savepoint_name;

-- 释放保存点
RELEASE SAVEPOINT savepoint_name;

-- 设置事务自动提交（默认开启）
SET autocommit = 1;  -- 开启自动提交
SET autocommit = 0;  -- 关闭自动提交（需要手动 COMMIT）
```

## 自动提交

```sql
-- 查看当前自动提交状态
SHOW VARIABLES LIKE 'autocommit';

-- 关闭自动提交后，每个 SQL 都是独立事务
SET autocommit = 0;

-- 执行 SQL（需要手动提交）
UPDATE accounts SET balance = balance - 100 WHERE user_id = 1;

-- 必须手动提交，否则其他会话看不到修改
COMMIT;

-- 也可以回滚
ROLLBACK;
```

## 隔离级别

MySQL 支持四种事务隔离级别，级别越高，数据一致性越好，但并发性能越差。

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|----------|------|-----------|------|
| READ UNCOMMITTED | ✅ 可能 | ✅ 可能 | ✅ 可能 |
| READ COMMITTED | ❌ 不可能 | ✅ 可能 | ✅ 可能 |
| REPEATABLE READ（默认） | ❌ 不可能 | ❌ 不可能 | ✅ 可能 |
| SERIALIZABLE | ❌ 不可能 | ❌ 不可能 | ❌ 不可能 |

### 1. READ UNCOMMITTED（读未提交）

最低隔离级别，事务可以读取其他未提交事务的修改。

```sql
-- 会话1：开启事务，修改数据（未提交）
START TRANSACTION;
UPDATE accounts SET balance = 1000 WHERE user_id = 1;

-- 会话2：设置隔离级别，读取（读到会话1未提交的数据）
SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
SELECT balance FROM accounts WHERE user_id = 1;  -- 读到 1000（脏读）

-- 会话1：回滚
ROLLBACK;

-- 会话2：再次查询
SELECT balance FROM accounts WHERE user_id = 1;  -- 读到 900（和之前不同）
```

### 2. READ COMMITTED（读已提交）

事务只能读取已提交事务的修改，但同一事务内多次读取可能得到不同结果（不可重复读）。

```sql
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 会话1：开启事务
START TRANSACTION;
SELECT balance FROM accounts WHERE user_id = 1;  -- 900

-- 会话2：开启事务，修改并提交
START TRANSACTION;
UPDATE accounts SET balance = 1000 WHERE user_id = 1;
COMMIT;

-- 会话1：再次查询（读到新值 1000，不可重复读）
SELECT balance FROM accounts WHERE user_id = 1;  -- 1000
```

### 3. REPEATABLE READ（可重复读）

MySQL 默认隔离级别。同一事务内多次读取结果一致，但可能出现幻读（同一查询返回了之前不存在的行）。

```sql
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 会话1：开启事务
START TRANSACTION;
SELECT balance FROM accounts WHERE user_id = 1;  -- 900

-- 会话2：插入新数据并提交
INSERT INTO accounts (user_id, balance) VALUES (3, 500);
COMMIT;

-- 会话1：再次查询（REPEATABLE READ 不会读到新行，除非使用锁或快照更新）
SELECT * FROM accounts;  -- 看不到会话2插入的新行（幻读问题被缓解，但可能存在）
```

### 4. SERIALIZABLE（串行化）

最高隔离级别，事务强制串行执行，完全避免幻读，但性能最差。

```sql
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;

START TRANSACTION;
SELECT * FROM accounts;
-- 其他事务的 INSERT 会被阻塞，直到当前事务结束
COMMIT;
```

### 设置隔离级别

```sql
-- 设置全局隔离级别（需重新连接生效）
SET GLOBAL transaction_isolation = 'REPEATABLE-READ';

-- 设置会话隔离级别
SET SESSION transaction_isolation = 'READ COMMITTED';

-- 设置下次事务隔离级别
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- 查看当前隔离级别
SHOW VARIABLES LIKE 'transaction_isolation';
```

## 事务隔离实现原理

### MVCC（多版本并发控制）

InnoDB 使用 MVCC 实现非阻塞读：

- 每行数据有两个隐藏列：`DB_TRX_ID`（最近修改的事务ID）、`DB_ROLL_PTR`（指向 undo log 的指针）
- 读取时根据事务快照判断可见性

```
事务A开始
事务B开始
事务B修改数据X，生成新版本
事务A读取X，看到的是旧版本（undo log）
事务B提交
事务A再次读取X，看到的还是旧版本（可重复读）
```

### 锁机制

```sql
-- 共享锁（S锁）：读取数据
SELECT * FROM users WHERE id = 1 LOCK IN SHARE MODE;

-- 排他锁（X锁）：修改数据
SELECT * FROM users WHERE id = 1 FOR UPDATE;

-- 表级锁
LOCK TABLES users WRITE;
UNLOCK TABLES;
```

## 事务与并发问题

### 1. 脏读（Dirty Read）

读取到其他事务未提交的修改。

### 2. 不可重复读（Non-repeatable Read）

同一事务内，两次读取同一行数据结果不同。

### 3. 幻读（Phantom Read）

同一事务内，两次查询结果集不同（多了或少了行）。

## 事务使用最佳实践

### 1. 短事务原则

事务时间越短越好，减少锁竞争。

```sql
-- ❌ 不好：事务中包含用户交互
START TRANSACTION;
UPDATE accounts SET balance = balance - 100 WHERE user_id = 1;
-- 用户思考或网络延迟...
COMMIT;

-- ✅ 好：快速完成
START TRANSACTION;
UPDATE accounts SET balance = balance - 100 WHERE user_id = 1;
UPDATE orders SET status = 'paid' WHERE order_id = 123;
COMMIT;
```

### 2. 选择合适隔离级别

```sql
-- 读多写少场景：REPEATABLE READ 或 READ COMMITTED
-- 金融转账：SERIALIZABLE 或 READ COMMITTED + 锁
-- 报表统计：READ COMMITTED
```

### 3. 合理使用保存点

```sql
START TRANSACTION;
INSERT INTO orders (...) VALUES (...);
SAVEPOINT order_inserted;

UPDATE inventory SET stock = stock - 1 WHERE product_id = 1;
-- 如果后面失败，可以回滚到保存点而不是整个事务
ROLLBACK TO SAVEPOINT order_inserted;

INSERT INTO payments (...) VALUES (...);
COMMIT;
```

### 4. 注意锁的范围

```sql
-- ❌ 行级锁未正确使用（无索引导致锁表）
START TRANSACTION;
UPDATE orders SET status = 'shipped' WHERE user_name = 'john';  -- user_name 无索引，锁全表
-- 其他事务全部阻塞！

-- ✅ 使用索引或主键
START TRANSACTION;
UPDATE orders SET status = 'shipped' WHERE user_id = 1;  -- user_id 有索引，只锁特定行
COMMIT;
```

### 5. 异常处理

```sql
-- Python 中使用事务
import pymysql

conn = pymysql.connect(...)
try:
    with conn.cursor() as cursor:
        cursor.execute("START TRANSACTION")
        cursor.execute("UPDATE accounts SET balance = balance - 100 WHERE user_id = 1")
        cursor.execute("UPDATE accounts SET balance = balance + 100 WHERE user_id = 2")
        cursor.execute("COMMIT")
except Exception as e:
    cursor.execute("ROLLBACK")
    print(f"Transaction failed: {e}")
finally:
    conn.close()
```

## 分布式事务

当事务涉及多个数据库时，使用分布式事务：

```sql
-- 两阶段提交（2PC）
XA START 'transaction_id';
UPDATE accounts SET balance = balance - 100 WHERE user_id = 1;
XA END 'transaction_id';
XA PREPARE 'transaction_id';
-- 检查其他参与者状态
XA COMMIT 'transaction_id';
-- 或 XA ROLLBACK 'transaction_id';
```

## 查看事务状态

```sql
-- 查看当前连接的事务
SHOW PROCESSLIST;

-- 查看锁等待
SHOW ENGINE INNODB STATUS;

-- 查看长事务
SELECT * FROM information_schema.INNODB_TRX 
WHERE trx_started < DATE_SUB(NOW(), INTERVAL 1 HOUR);
```
